# 台灣農產品產銷預測系統：基於多模型集成學習之智慧農業決策支援平台

---

## 第一章　研究背景與動機

### 1.1 台灣農業產銷現況與挑戰

台灣地處亞熱帶與熱帶交界，四季分明且雨量充沛，具備優越的農業生產條件。根據農業部統計，全台設有 20 個主要蔬果批發市場，涵蓋 22 個縣市行政區域，每年處理數百萬筆農產品交易紀錄。然而，農產品價格波動劇烈、供需失衡的問題長期困擾產業各環節的參與者——從產地農民到批發市場，再到終端消費者。

農產品市場具有以下顯著特性，使其價格預測面臨極大挑戰：

| 挑戰 | 現象 | 對產銷的影響 |
|------|------|------------|
| **強季節性波動** | 不同作物受生長週期影響，產量與價格呈現明顯的年度季節性模式（如香蕉 5–9 月盛產） | 供給增加時價格下跌，冬季供給減少時價格走高，年度循環波動劇烈 |
| **極端天氣衝擊** | 台灣每年平均遭受 3–5 個颱風侵襲，單月降雨量常超過 300mm | 颱風直接損害農作物，並在颱風後 1–2 個月內引發價格劇烈波動 |
| **多重市場交互作用** | 全台 20 個批發市場之間存在價格傳導效應 | 北部市場（如台北一、二市）的價格波動牽動全國市場走勢 |
| **資訊不對稱** | 農民決定種植面積時，缺乏未來供需的前瞻性預測資訊 | 反覆陷入「價格好→搶種→供過於求→價格崩跌」的惡性循環 |

### 1.2 智慧農業與數據驅動決策的趨勢

近年來，全球農業正經歷數位轉型的浪潮。聯合國糧農組織（FAO）與世界銀行均強調，運用人工智慧與大數據分析技術輔助農業決策，是提升糧食安全與農民收益的關鍵途徑。在此背景下，「智慧農業」（Smart Agriculture）已成為各國農業政策的核心方向。

台灣農業部自 2017 年起推動「智慧農業 4.0」計畫，鼓勵將物聯網（IoT）、人工智慧（AI）及大數據技術應用於農業生產管理。農委會農糧署亦建置農產品交易行情系統（AMIS），提供每日交易資料的開放存取。中央氣象署（CWA）同步開放氣象觀測資料 API，為農業氣象分析提供資料基礎。

然而，現有的公開資料平台多停留在「查詢歷史資料」的階段，缺乏前瞻性的預測分析能力。農民與市場管理者仍主要依賴經驗法則進行決策，對科學化的產銷預測工具有迫切需求。

### 1.3 研究動機

本研究的動機源於以下三項觀察：

| 動機 | 現況缺口 | 本研究回應 |
|------|---------|----------|
| **多源異質資料的整合需求** | 農業資料分散於農業部（民國紀年）、氣象署（西元紀年）、各縣市統計等不同來源，格式不一且粒度各異（日度交易 vs. 年度產量），缺乏整合方案 | 建立自動化的多源資料擷取、轉換與入庫管道，統一日期格式與時間粒度，形成綜合分析資料庫 |
| **單一模型的預測侷限** | 農產品價格受長期趨勢、季節週期、短期波動及外生衝擊（颱風）等多重因素影響。ARIMA 擅長序列相關性但缺乏非線性能力；XGBoost 善於特徵交互但趨勢建模較弱——單一模型無法全面捕獲 | 結合 Prophet、SARIMA、XGBoost、LightGBM 四種模型的互補優勢，以反向 MAPE 加權建構穩健的集成預測框架 |
| **預測結果的實務應用落地** | 學術預測模型多停留在論文層面，缺乏可供農民、市場管理者及政策制定者直接使用的互動式工具 | 開發完整的 Web 應用系統，以直覺化圖表、台灣地圖與預警燈號呈現預測結果，實現研究成果的實務轉化 |

### 1.4 研究對象：五種關鍵蔬菜的選定

台灣蔬菜種類繁多，為聚焦研究資源並產出具代表性的實證結果，本研究經系統化篩選後，最終選定以下五種蔬菜作為核心研究對象：

| 作物 | 英文名 | 旺季月份 | 淡季月份 | 選定理由 |
|------|--------|---------|---------|---------|
| **甘藍（高麗菜）** | Cabbage | 11月–3月 | 6月–8月 | 台灣產量最大的葉菜類，價格波動最受社會關注，歷年搶種導致價格崩跌的案例最為典型 |
| **萵苣（A菜/大陸妹）** | Lettuce | 10月–3月 | 6月–8月 | 消費量穩定成長的葉菜類，夏季高溫減產明顯，價格季節性波動幅度大 |
| **小白菜** | Bok Choy | 10月–3月 | 6月–8月 | 台灣家庭最常購買的平價蔬菜之一，生長週期短（約 25-30 天），對短期天氣變化反應最靈敏 |
| **花椰菜** | Cauliflower | 11月–3月 | 6月–9月 | 冬季主力蔬菜，近年因健康飲食趨勢需求攀升，淡季為全年最長（4 個月），供需落差顯著 |
| **青蔥** | Green Onion | 10月–2月 | 6月–8月 | 台灣料理中不可或缺的辛香蔬菜，颱風後價格暴漲最為劇烈（曾出現 5 至 10 倍漲幅），是颱風衝擊研究的最佳標的 |

**選定原則：**

1. **民生代表性**：五種蔬菜均為台灣消費者日常高頻購買品項，涵蓋葉菜（甘藍、萵苣、小白菜）、花菜（花椰菜）及辛香料（青蔥）三種食用類型，價格波動直接影響民生消費支出。

2. **季節性特徵一致與差異兼具**：五種蔬菜的旺季均集中於秋冬季（10月–3月），淡季集中於夏季（6月–8月），具有共通的季節性模式（seasonality_mode 均為 additive），利於模型架構統一設計。但各作物的旺季起始月份與淡季長度存在差異（如花椰菜淡季長達 4 個月，而甘藍淡季僅 3 個月），可驗證模型對不同季節模式的適應性。

3. **颱風敏感度差異**：青蔥因根系淺、生長於露天環境，對颱風衝擊最為敏感，是驗證多維度颱風特徵工程效果的理想作物；而甘藍因種植區域集中於高海拔山區（如梨山），颱風時的運輸中斷亦是關鍵價格推動因素。此差異可驗證颱風特徵在不同受災機制下的預測改善效果。

4. **資料充足性**：五種蔬菜在農業部 AMIS 系統中均有長期、連續的每日交易紀錄，資料量足以支撐時間序列模型的訓練需求（月度資料點超過 60 個以上）。

5. **政策與產業關注度**：甘藍與青蔥是農業部「蔬菜產銷預警」機制中的重點監測品項，其價格異常波動常引發媒體報導與政策介入（如釋出庫存、啟動保價收購），研究成果具有直接的政策參考價值。

## 第二章　文獻回顧與研究缺口

### 2.1 農產品價格預測相關研究

農產品價格預測是農業經濟學與時間序列分析的重要研究領域。根據預測方法的演進，相關文獻可歸納為以下四個階段：

#### 2.1.1 傳統統計方法

早期研究主要採用自迴歸移動平均（ARIMA）族系模型進行農產品價格預測。Box 與 Jenkins（1970）提出的 ARIMA 建模方法論，至今仍是時間序列分析的基礎。Hyndman 與 Khandakar（2008）發展的自動 ARIMA 參數選擇演算法（auto.arima），大幅降低了 ARIMA 建模的門檻。針對具有明顯季節性的農產品價格序列，季節性 ARIMA（SARIMA）成為廣泛採用的工具，其透過季節性差分與季節性自迴歸項，直接建模年度週期性波動。

然而，傳統 ARIMA 模型假設線性結構，難以捕捉農產品價格中常見的非線性動態，如颱風衝擊引發的價格跳升、供需結構性轉變等。

#### 2.1.2 機器學習方法

隨著機器學習技術的發展，梯度提升樹（Gradient Boosting）族系模型在農產品價格預測中展現出良好的效果。Chen 與 Guestrin（2016）提出的 XGBoost，透過正則化目標函數與高效的分裂演算法，在各類預測競賽中取得優異成績。Ke 等人（2017）提出的 LightGBM，採用基於梯度的單邊採樣（GOSS）與互斥特徵捆綁（EFB）技術，在大規模資料集上顯著提升訓練效率。

機器學習方法的核心優勢在於特徵工程的靈活性——研究者可將滯後值（lag features）、滾動統計量、日曆特徵、氣象變數等多種資訊納入模型，學習複雜的非線性交互效應。然而，純機器學習方法對時間序列的趨勢成分與季節性成分的建模不如專門的時間序列模型直觀。

#### 2.1.3 可分解式預測模型

Facebook（現 Meta）於 2018 年發表的 Prophet 模型，採用可加或可乘式分解架構，將時間序列拆解為趨勢、季節性、假日效應與外生回歸項等成分。Prophet 的設計哲學是「讓分析師更容易產出合理的預測」，其自動化的變化點偵測（changepoint detection）與彈性的季節性建模，特別適合處理具有明顯年度週期性的農產品資料。

Prophet 的限制在於其本質上仍是曲線配適（curve fitting）方法，對短期序列相關性的捕獲能力不如 ARIMA 族系模型。

#### 2.1.4 集成學習方法

模型集成（Ensemble Learning）是提升預測穩健性的重要策略。常見的集成方法包括簡單平均、加權平均及堆疊（Stacking）。在農產品預測領域，部分研究採用反向預測誤差加權（Inverse Error Weighting）策略，根據各模型在驗證集上的表現動態分配權重，使表現較佳的模型獲得更高的影響力。

相較於等權平均，反向誤差加權的優勢在於：（1）自動適應不同作物、不同時期的最佳模型組合；（2）避免表現較差的模型拖累整體預測精度；（3）權重計算透明，便於解釋。

#### 2.1.5 四類預測方法綜合比較

| 方法類別 | 代表方法/文獻 | 核心原理 | 優勢 | 限制 |
|---------|-------------|---------|------|------|
| 傳統統計 | ARIMA / SARIMA (Box & Jenkins, 1970; Hyndman, 2008) | 自迴歸與移動平均建模序列相關性，季節差分處理週期 | 理論成熟、可解釋性高、小樣本表現穩定 | 假設線性結構，難以捕獲非線性動態與外生衝擊 |
| 機器學習 | XGBoost (Chen, 2016) / LightGBM (Ke, 2017) | 梯度提升樹學習特徵交互，正則化防止過擬合 | 特徵工程靈活、可納入多種外生變數、非線性建模能力強 | 對時間序列的趨勢與季節性成分建模不如專門模型直觀 |
| 可分解式 | Prophet (Meta, 2018) | 將時間序列分解為趨勢+季節性+假日+回歸項 | 自動變化點偵測、季節性建模彈性高、易於使用 | 本質為曲線配適，對短期序列相關性捕獲能力較弱 |
| 集成學習 | 反向誤差加權 / Stacking | 結合多模型預測，依驗證表現動態分配權重 | 提升穩健性、自動適應最佳模型組合、權重透明 | 需訓練多個模型，計算成本較高 |

### 2.2 氣象因子與農產品價格關聯研究

氣象條件對農產品供給的影響已有豐富的實證研究。溫度、降雨量、濕度等氣象變數直接影響作物生長與產量，進而傳導至市場價格。針對台灣特殊的颱風風險，相關研究發現颱風對蔬菜價格的衝擊通常具有 1 至 2 個月的延遲效應——颱風當月因搶收而短暫壓低價格，之後因供給短缺而推高價格，約 2 至 3 個月後逐漸回歸正常。

然而，現有研究多將颱風影響視為單一二元變數（有/無颱風），缺乏對颱風強度等級（輕度/中度/強度）、影響路徑及受影響縣市的精細建模。

### 2.3 農業決策支援系統

農業決策支援系統（Agricultural Decision Support Systems, ADSS）的發展可追溯至 1980 年代的專家系統。現代 ADSS 強調資料驅動、視覺化導向與使用者友善的設計理念。然而，文獻中多數 ADSS 研究停留在概念架構或原型系統階段，缺乏完整的前後端實現與實際的資料整合。

### 2.4 研究缺口

綜合文獻回顧，本研究識別出以下研究缺口：

| 缺口編號 | 描述 | 本研究回應 |
|---------|------|----------|
| G1 | 既有研究多採用單一模型，缺乏系統化的多模型集成框架 | 建構 Prophet + SARIMA + XGBoost + LightGBM 四模型集成架構，採用反向 MAPE 加權策略 |
| G2 | 颱風影響的建模過於粗略（僅有/無二元變數） | 設計多維度颱風特徵：強度等級、颱風後 1/2 個月旗標、距上次颱風天數、極端降雨旗標 |
| G3 | 多源異質資料（交易、氣象、產量、颱風）的整合方法缺乏系統化論述 | 建立統一的資料管道，處理民國/西元轉換、日度/月度對齊、多站點聚合等異質性問題 |
| G4 | 預測模型與實務應用之間的落差——研究成果難以轉化為可用工具 | 開發完整的 Web 應用系統，包含互動式圖表、地圖視覺化、資料品質監控與自動排程 |
| G5 | 缺乏針對台灣 22 種主要作物的大規模系統化預測研究 | 涵蓋 22 種作物（含水果、蔬菜、根莖、穀物四大類），建立標準化的作物配置檔架構 |

## 第三章　研究目的與問題

### 3.1 待解決的問題與服務對象

#### 3.1.1 核心問題

台灣蔬菜產銷體系長期面臨「菜金菜土」的結構性困境——蔬菜價格在短期內大幅震盪，供需雙方均深受其害。具體而言，本研究欲解決以下三層問題：

**問題一：產銷資訊斷層。** 蔬菜從產地到市場的供需資訊缺乏即時、前瞻性的預測工具。農民在決定種植面積時，僅能參考「當下」或「上一期」的市場行情，無法預見未來 1 至 6 個月的市場走勢，導致「看到高價就搶種，收成時價格已崩」的惡性循環反覆發生。

**問題二：極端天氣應變不足。** 台灣每年颱風季（6 至 10 月）對蔬菜供給造成劇烈衝擊，但現有工具無法量化「不同強度的颱風，對不同蔬菜品項，在颱風後 1 至 2 個月內的價格影響幅度」。市場管理者與政策制定者在颱風來臨前缺乏科學化的情境推估依據。

**問題三：決策工具缺乏。** 現有農業公開平台（如農業部 AMIS）僅提供歷史資料查詢功能，不具備預測分析、視覺化比較與預警通知能力。學術研究中的預測模型多停留在論文階段，未能轉化為可供產業使用者直接操作的工具。

#### 3.1.2 服務對象

本系統的設計針對以下三類使用者，各自對應不同的功能需求：

| 服務對象 | 角色描述 | 核心需求 | 系統對應功能 |
|---------|---------|---------|------------|
| **蔬菜農民** | 需決定種植品項與面積的第一線生產者 | 「未來 1-3 個月，我種的菜價格會漲還是跌？」 | 價格預測趨勢圖、產銷預警燈號（紅黃綠）、季節性分析圖 |
| **批發市場管理者** | 全台 20 個蔬果批發市場的調度與管理人員 | 「颱風來臨前，哪些蔬菜可能短缺？各縣市供給量如何分佈？」 | 颱風情境模擬、台灣縣市地圖、跨市場比較分析、交易量預測 |
| **農業政策制定者** | 農業部、各縣市農業局的政策規劃人員 | 「目前哪些品項有供過於求的風險？是否需要啟動保價收購？」 | 多品項儀表板總覽、歷史趨勢與預測對照、資料品質監控、匯出報表 |

### 3.2 研究目的

本研究旨在以五種台灣關鍵蔬菜（甘藍、萵苣、小白菜、花椰菜、青蔥）為研究對象，設計並實現一套**基於多模型集成學習的蔬菜產銷預測與決策支援平台**。具體研究目的如下：

| 研究目的 | 具體內容 | 對應 RQ |
|---------|---------|--------|
| **目的一：建立多源農業資料整合管道** | 整合農業部 AMIS 交易系統、中央氣象署 CWA 觀測資料、颱風歷史資料庫及農業生產統計等四大外部資料源，建立自動化的資料擷取、清洗與入庫流程，解決民國/西元紀年差異、日度/月度粒度不一致等異質性問題 | RQ1 |
| **目的二：發展多模型集成預測框架** | 設計包含 Prophet、SARIMA、XGBoost 與 LightGBM 四種預測模型的集成架構，透過反向 MAPE 加權策略自動為每種蔬菜分配最佳模型組合權重，提供 1/3/6 個月的價格與交易量預測 | RQ2 |
| **目的三：實現可落地的互動式決策支援平台** | 開發完整的 Web 應用系統，包含互動式儀表板、台灣縣市地圖、颱風情境模擬、產銷預警燈號與資料品質監控，使預測成果能直接服務於農民、市場管理者及政策制定者三類使用者 | RQ3 |

### 3.3 研究問題

基於上述問題意識與研究目的，本研究提出以下三組研究問題：

---

**RQ1：如何有效整合多源異質農業資料，為五種蔬菜建立完整的產銷分析資料庫？**

> **要解決的問題：** 台灣農業資料分散於多個政府機關，格式不一（民國/西元紀年）、粒度各異（日度交易 vs. 年度產量）、空間對應不明確（氣象站 vs. 行政區域），目前沒有現成的整合方案。
>
> **對象：** 農業資料工程與系統維運層面。

- 子問題 1a：如何處理農業部（民國紀年）與氣象署（西元紀年）之間的日期格式差異，並將日度交易、日度氣象、月度/年度產量資料對齊至統一的月度時間尺度？
- 子問題 1b：如何建立 22 個氣象站與 22 個縣市的空間對應關係，以支撐縣市層級的預測分析？
- 子問題 1c：如何設計資料品質監控機制（紅黃綠燈號），幫助使用者判斷預測結果的可信度？

---

**RQ2：四模型集成預測架構（Prophet + SARIMA + XGBoost + LightGBM）在五種蔬菜的價格與交易量預測上表現為何？**

> **要解決的問題：** 單一模型無法全面捕獲蔬菜價格的複雜動態——統計模型擅長趨勢與季節性，但對非線性衝擊反應不足；機器學習模型善於學習特徵交互，但對時序結構建模不如專門模型。需要驗證集成策略能否結合各模型優勢，穩定提升預測精度。
>
> **對象：** 五種目標蔬菜的月度均價（price_avg）與月度交易量（volume）。

- 子問題 2a：四種基礎模型各自在五種蔬菜上的預測特性與適用情境為何？是否存在「某類模型始終較優」的模式？
- 子問題 2b：反向 MAPE 加權集成策略相較於單一最佳模型與等權平均，是否能在多數作物上顯著提升預測精度（以 MAPE ≤ 15%、R² ≥ 0.60 為目標）？
- 子問題 2c：多維度颱風特徵（強度等級、颱風後 1/2 個月延遲效應、極端降雨旗標）相較於傳統二元颱風變數，對青蔥、甘藍等颱風敏感作物的預測改善幅度為何？

---

**RQ3：如何設計一套符合農民、市場管理者及政策制定者需求的互動式決策支援平台？**

> **要解決的問題：** 預測模型的學術價值需要透過直覺化的介面才能轉化為實務決策依據。不同使用者（農民看價格趨勢、管理者看區域分佈、政策者看全局概覽）有不同的資訊需求與操作習慣，需要差異化的功能設計。
>
> **對象：** 三類終端使用者——蔬菜農民、批發市場管理者、農業政策制定者。

- 子問題 3a：颱風情境模擬功能（可調整颱風強度與月份）如何幫助市場管理者在颱風季前進行預判與調度？
- 子問題 3b：台灣縣市地圖視覺化與多層次篩選功能（作物/時間/指標/縣市），如何提升政策制定者的分析效率？
- 子問題 3c：產銷預警燈號系統（供給指數、價格跌幅、面積成長率三項指標）如何幫助農民及早識別供過於求風險？

---

### 4.4 多模型集成預測框架

#### 4.4.1 框架概覽

本系統採用異質模型集成策略，結合四種具有不同預測特性的模型：

| 模型 | 類型 | 擅長捕獲的模式 | 核心參數 |
|------|------|---------------|---------|
| Prophet | 可分解式 | 長期趨勢、年度季節性、變化點 | seasonality_mode, changepoint_prior_scale |
| SARIMA | 統計時序 | 序列相關性、短期波動、季節差分 | (p,d,q)(P,D,Q)m，自動搜尋 (AIC) |
| XGBoost | 梯度提升樹 | 非線性特徵交互、多變數關聯 | n_estimators=500, max_depth=6, lr=0.05 |
| LightGBM | 梯度提升樹 | 大規模特徵、高效訓練、類別變數 | GOSS 採樣、EFB 特徵捆綁 |

#### 4.4.2 特徵工程體系

```
原始月度時間序列
  │
  ├── 時序滯後特徵 (lag_1, lag_2, lag_3, lag_6, lag_12)
  │
  ├── 滾動統計特徵 (roll_mean, roll_std, roll_max, roll_min)
  │     └── 視窗大小: 3, 6, 12 個月
  │
  ├── 日曆特徵 (month, quarter, is_peak_season)
  │     └── 三角函數編碼: month_sin, month_cos
  │
  ├── 年度對比特徵 (yoy_value, yoy_growth)
  │
  ├── 氣象特徵 (temp_avg, rainfall_mm, temp_anomaly)
  │     └── 來源: CWA 22 個氣象站月度聚合
  │
  └── 颱風特徵 (多維度)
        ├── is_typhoon_month (該月是否有颱風警報)
        ├── typhoon_count (該月颱風數量)
        ├── typhoon_intensity_max (最大強度: 1=輕度/2=中度/3=強度)
        ├── days_since_typhoon (距上次颱風天數, 上限 365)
        ├── post_typhoon_1m (颱風結束後 0-30 天旗標)

        └── extreme_rainfall_flag (月降雨量 > 300mm)
```

#### 4.4.3 集成權重計算

採用反向 MAPE 加權策略，根據各模型在驗證集上的預測表現動態分配權重：

```
                    1 / MAPE_i
    weight_i = ─────────────────
               Σ_j (1 / MAPE_j)
```

**集成預測值計算：**
- 點估計：`ŷ_ensemble = Σ(w_i × ŷ_i)`
- 信賴區間下界：`lower = min(lower_i)`（取最保守邊界）
- 信賴區間上界：`upper = max(upper_i)`（取最保守邊界）

**範例：**
若驗證集結果為 Prophet MAPE=5.2%、SARIMA MAPE=7.1%、XGBoost MAPE=4.8%、LightGBM MAPE=5.5%，
則權重分別為 35.5%、26.0%、38.5%、33.6%（歸一化後）。

#### 4.4.4 訓練與評估流程

```
完整預測管道 (run_full_pipeline)
│
├── 載入 22 個作物配置
│
└── 對每個作物 × 每個指標 (price_avg, volume) 執行:
    │
    ├── 1. 資料查詢
    │   ├── TradingData → 月度聚合
    │   ├── WeatherData → 月度聚合
    │   └── TyphoonEvent → 事件匹配
    │
    ├── 2. 資料前處理
    │   ├── 缺失值填充 (向前/向後)
    │   ├── 異常值裁剪 (IQR 法)
    │   └── 排除最近 90 天 (holdout)
    │
    ├── 3. 訓練/驗證分割 (70/30)
    │
    ├── 4. 四模型獨立訓練
    │   ├── Prophet (含外生回歸)
    │   ├── SARIMA (自動參數搜尋)

    │   └── 計算 MSE, RMSE, MAE, R², MAPE
    │
    ├── 6. 集成預測
```

### 4.7 資料品質監控機制

系統內建三級燈號資料品質監控：

| 燈號 | 覆蓋率 | 含義 | 建議行動 |
|------|--------|------|---------|
| 🟢 綠燈 | ≥ 90% | 資料品質良好，預測可信度高 | 正常使用 |
| 🟡 黃燈 | 70%-89% | 資料品質尚可，預測結果需謹慎參考 | 關注缺失區段 |
| 🔴 紅燈 | < 70% | 資料品質不足，預測可信度較低 | 優先補齊資料 |

監控面向涵蓋：交易資料完整性、氣象資料覆蓋率、生產統計更新頻率，以及各作物的資料可用月數。

## 第五章　工作內容重點

### 5.1 工作項目

本研究的工作內容劃分為以下五大模組：

#### 模組一：多源資料整合平台

| 工作項目 | 具體內容 |
|---------|---------|
| 1-1 AMIS 交易資料串接 | 開發農業部 API 連接器，處理民國紀年轉換，支援日期範圍批量擷取（速率限制 1.0 秒/次），建立作物名稱模糊匹配機制 |
| 1-2 CWA 氣象資料串接 | 開發氣象署 API 連接器，建立 22 個氣象站與縣市的對應表，實現多站點聚合為縣級月度統計 |
| 1-3 颱風歷史資料庫 | 匯入中央氣象署颱風資料庫歷史事件（145 筆），建立颱風事件與受影響縣市的關聯表 |
| 1-4 生產統計匯入 | 開發多格式 CSV 匯入器，支援中英文欄位名稱自動映射，建立年度/月度生產統計資料庫 |
| 1-5 資料品質監控 | 實作三級燈號監控系統，涵蓋覆蓋率、完整性與時效性三個維度 |

#### 模組二：預測模型開發

| 工作項目 | 具體內容 |
|---------|---------|
| 2-1 Prophet 預測器 | 實現可加/可乘季節性模式，支援峰季回歸變數與氣象外生變數，自動變化點偵測 |
| 2-2 SARIMA 預測器 | 整合 pmdarima 自動參數搜尋，自適應降級機制（資料不足時退化為非季節 ARIMA） |
| 2-3 XGBoost 預測器 | 實現自迴歸多步預測策略，整合完整特徵工程管道，支援分位數回歸信賴區間 |
| 2-4 LightGBM 預測器 | 利用 GOSS 與 EFB 加速訓練，整合與 XGBoost 相同的特徵工程體系 |
| 2-5 集成預測引擎 | 實現反向 MAPE 加權集成，動態權重計算，保守信賴區間合成 |
| 2-6 模型評估框架 | 實現展開視窗交叉驗證（min_train=24月, n_folds=3），五項評估指標 |

#### 模組三：特徵工程與資料前處理

| 工作項目 | 具體內容 |
|---------|---------|
| 3-1 時序特徵生成 | 滯後特徵（1/2/3/6/12 月）、滾動統計（3/6/12 月視窗的均值/標準差/最大/最小） |
| 3-2 日曆與季節特徵 | 月份、季度、峰季旗標、三角函數季節編碼 |
| 3-3 氣象特徵整合 | 縣市月均溫、月累積降雨、溫度距平值 |
| 3-4 颱風特徵設計 | 七維颱風特徵向量（詳見 4.4.2 節） |
| 3-5 資料前處理管道 | 日度→月度重取樣、缺失值填充、IQR 異常值裁剪、MinMax 縮放 |

#### 模組四：Web 應用系統開發

| 工作項目 | 具體內容 |
|---------|---------|
| 4-1 後端 API 開發 | 37 個 RESTful 端點，涵蓋預測查詢、交易分析、資料同步、匯出等功能 |
| 4-2 儀表板頁面 | 摘要卡片、時間序列圖、台灣地圖、市場排行、警示列表 |
| 4-3 預測分析頁面 | 預測結果卡片、歷史+預測圖、模型對比、特徵重要性、信心度指標 |
| 4-4 交易分析頁面 | 價量雙軸圖、季節性分析、市場比較、交易明細表 |
| 4-5 資料管理頁面 | 同步狀態監控、四步驟上傳精靈、模型重訓觸發 |
| 4-6 颱風情境模擬 | 颱風強度 × 月份互動式模擬，顯示預測調整幅度 |

#### 模組五：自動化運維

| 工作項目 | 具體內容 |
|---------|---------|
| 5-1 排程管理 | APScheduler 四項定時任務（每日資料擷取、每週模型重訓、每月資料清理） |
| 5-2 作物配置管理 | 22 個作物的 JSON 配置檔，包含模型參數、季節性設定、特徵選擇 |
| 5-3 模型版本管理 | ModelRegistry 表記錄每次訓練的指標、檔案路徑與特徵重要性 |

### 5.2 預期貢獻

#### 學術貢獻

1. **多模型集成框架的實證驗證**：本研究系統性地比較 Prophet、SARIMA、XGBoost 與 LightGBM 四種模型在台灣 22 種農產品上的預測表現，提供跨模型、跨作物的大規模實證結果，補充現有文獻中缺乏的系統化比較研究。

2. **農業專用颱風特徵工程**：提出一套七維颱風特徵向量設計，將颱風對農產品價格的衝擊從傳統的二元變數擴展為包含強度、延遲效應及極端降雨的多維度建模，為農業氣象研究提供可複用的特徵工程方法。

3. **多源異質資料整合方法論**：提出台灣農業多源資料（交易、氣象、產量、颱風）的標準化整合流程，包含日期格式轉換、時間粒度對齊、空間對應等技術，為後續研究提供可參考的資料工程實踐。

#### 實務貢獻

4. **農民決策支援**：提供直覺化的價格趨勢預測與產銷預警燈號，幫助農民在種植決策階段獲取市場前瞻資訊，降低因資訊不對稱導致的盲目搶種風險。

5. **市場管理者工具**：提供跨市場、跨縣市的比較分析功能與颱風情境模擬，協助批發市場管理者提前備妥調度方案。

6. **政策參考依據**：系統產出的預測結果與歷史分析可作為農業政策制定的量化參考，如調節性釋出庫存、啟動價格穩定機制等。

### 5.3 關鍵績效指標（KPI）

#### 系統建置 KPI

| KPI 項目 | 目標值 | 衡量方式 |
|---------|--------|---------|
| 資料源整合數量 | ≥ 4 個外部資料源 | AMIS、CWA、颱風DB、生產統計 |
| 核心研究蔬菜 | 5 種 | 甘藍、萵苣、小白菜、花椰菜、青蔥 |
| 涵蓋批發市場 | ≥ 20 個 | 全台主要蔬果批發市場 |
| 涵蓋縣市 | 22 個 | 含離島 |
| API 端點數量 | ≥ 37 個 | RESTful API 完整覆蓋 |
| 前端頁面/組件 | 5 頁 / ≥ 35 組件 | 含圖表、地圖、互動組件 |
| 資料庫總資料量 | ≥ 1,300 萬筆 | 交易 + 氣象 + 產量 + 颱風 |

#### 模型效能 KPI

| KPI 項目 | 目標值 | 說明 |
|---------|--------|------|
| 集成模型 MAPE | ≤ 15% | 五種蔬菜月均價預測的平均絕對百分比誤差 |
| 集成優於單一模型比例 | ≥ 70% | 在多數作物上，集成應優於任一單一模型 |
| R² (決定係數) | ≥ 0.60 | 模型對價格變異的解釋力 |
| 預測覆蓋率 | 100% | 5 種蔬菜 × 2 指標 × 3 期間皆有預測 |

#### 系統運維 KPI

| KPI 項目 | 目標值 | 說明 |
|---------|--------|------|
| 每日資料同步成功率 | ≥ 95% | AMIS + CWA 資料擷取穩定性 |
| 每週模型重訓完成率 | 100% | 排程任務按時執行 |
| API 回應時間 | ≤ 500ms | 主要查詢端點的 P95 延遲 |
| 資料品質綠燈比例 | ≥ 80% | 各作物資料品質達到「良好」等級 |
